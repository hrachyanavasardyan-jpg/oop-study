Topic 01
Enums & For-Each Loops
Enumerated types give names to fixed sets of values. For-each loops make iterating over arrays and collections cleaner and safer.
💡 Simple Explanation ▼

An enum (enumerated type) is like a list of named constants — a fixed set of values that a variable can hold. Instead of using magic numbers like 1 for Monday, you use WorkDay.MONDAY.

The for-each loop (enhanced for loop) lets you loop through every element in an array or collection without managing an index variable — cleaner, less error-prone.

🌍 Real-World Analogy
Enum = Traffic light colours. A traffic light can only ever be RED, YELLOW, or GREEN. You wouldn't use random numbers to represent that — you'd use named constants. That's an enum.

For-each = Reading every page of a book. You don't say "read page 0, read page 1, read page 2…" — you just say "for every page, read it."

Enums in Java are actually classes. That means they can have instance variables, constructors, and methods — making them very powerful.

☕ What Happens in Java ▼
  • Declared with enum Name { VALUE1, VALUE2, ... }
  • Each value is automatically an int ordinal starting from 0
  • Auto-provided methods: ordinal(), name(), values(), valueOf(), compareTo(), equals()
  • Can have a private instance variable + constructor to store extra data
  • For-each syntax: for (Type var : collection) { ... }
  • For-each cannot modify array elements (read-only access to element)
📝 Code from Class (Cleaned) ▼
Month Enum with Index
// Enum is a special class — each constant is an instance
enum Month {
    JANUARY, FEBRUARY, MARCH, APRIL, MAY, JUNE,
    JULY, AUGUST, SEPTEMBER, OCTOBER, NOVEMBER, DECEMBER;

    private String index; // "01", "02", ..., "12"

    // Constructor runs once per constant when class loads
    Month() {
        int val = ordinal() + 1; // ordinal() = 0-based position
        index = val < 10 ? "0" + val : "" + val;
    }

    // Static helper: converts int day to "DD" format
    public static String intToString(int v) {
        return v < 10 ? "0" + v : "" + v;
    }
}

// Usage: convert Date to MM/DD/YYYY format
Date date1 = new Date("May", 8, 1975);
String mm = Month.valueOf(date1.getMonth().toUpperCase()).index;
String dd = Month.intToString(date1.getDay());
System.out.println(mm + "/" + dd + "/" + date1.getYear());
// Output: 05/08/1975

// For-each vs regular loop — both do the same thing
double[] a = {1.5, 2.0, 3.5};

// Old way:
for (int i = 0; i < a.length; i++)
    System.out.println(a[i]);

// For-each (cleaner, preferred):
for (double element : a)
    System.out.println(element);
🔢 Step-by-Step Code Flow ▼
  1. JVM loads the Month enum class and creates 12 objects — one per constant.
  2. For each object, the constructor runs: ordinal() returns 0 for JANUARY, 1 for FEBRUARY, etc.
  3. val = ordinal() + 1 shifts the index to 1–12. The ternary pads single digits with a leading zero.
  4. Month.valueOf("MAY") looks up the constant named "MAY" and returns it.
  5. We access its .index field → "05".
  6. intToString(8) returns "08". String concatenation builds "05/08/1975".
⚠️ Common Mistakes ▼
❌ Using ordinal() as the month number

Month.JANUARY.ordinal() returns 0, not 1. Students forget ordinals are 0-based.

✅ Always add 1: ordinal() + 1 to get the real month number.
❌ Passing lowercase to valueOf()

Month.valueOf("may") throws IllegalArgumentException. Enum names are case-sensitive.

✅ Always call .toUpperCase() first: Month.valueOf(str.toUpperCase())
❌ Trying to modify array elements in for-each

for (int x : arr) { x = 0; } — this does NOT change the array! x is a copy.

✅ Use a regular indexed loop if you need to modify: for (int i = 0; i < arr.length; i++) arr[i] = 0;
Topic 02
Classes & Objects
A class is a blueprint. An object is a real thing built from that blueprint. This is the foundation of all OOP.
💡 Simple Explanation ▼

A class defines what something IS (its data) and what it can DO (its methods). An object is a specific instance of a class — built using the new keyword.

🌍 Real-World Analogy
Class = Cookie cutter. An object = the actual cookie. The cutter (class) defines the shape. Each cookie (object) is a separate thing with its own "dough" (data). You can make a thousand cookies from one cutter — each independent.

Every object has its own copy of instance variables (the data). Methods are shared — they just operate on whichever object called them using this.

☕ Key Java Concepts ▼
  • Constructor: Special method with the same name as the class. No return type. Called by new.
  • this(): Calls another constructor in the same class (must be first line)
  • Accessor (getter): Returns a private field's value
  • Mutator (setter): Changes a private field's value
  • toString(): Called automatically by System.out.println(obj)
  • equals(Object o): Override this — never use == to compare objects by value
📝 Best Practice: Date Class ▼
Date.java — Clean version from lectures
public class Date {

    // Private fields — only accessible inside this class
    private String month;
    private int day;
    private int year;

    // Default constructor — calls the full constructor via this()
    public Date() {
        this("January", 1, 0); // Must be FIRST line
    }

    // Full constructor
    public Date(String month, int day, int year) {
        this.month = month; // "this.month" = field; "month" = param
        this.day = day;
        this.year = year;
    }

    // Copy constructor — creates an independent copy
    public Date(Date that) {
        this(that.month, that.day, that.year);
    }

    // Getters (accessors)
    public String getMonth() { return month; }
    public int getDay()       { return day; }
    public int getYear()      { return year; }

    // Setters (mutators)
    public void setMonth(String month) { this.month = month; }
    public void setDay(int day)         { this.day = day; }
    public void setYear(int year)       { this.year = year; }

    // Returns a NEW Date one year later (doesn't modify this)
    public Date nextYear() {
        return new Date(month, day, year + 1);
    }

    // toString — called by System.out.println(date)
    public String toString() {
        return month + " " + day + ", " + year;
    }

    // equals — compares by VALUE, not memory address
    public boolean equals(Object object) {
        if (this == object) return true;   // same object?
        if (object == null) return false;  // null check
        if (this.getClass() != object.getClass()) return false;
        Date other = (Date) object;        // downcast safely
        return day == other.day && year == other.year
            && month.equals(other.month);
    }
}
⚠️ Common Mistakes ▼
❌ Using == to compare objects

date1 == date2 checks if they are the same object in memory, not if they have equal values. Always returns false for two separately created objects.

✅ Use .equals(): date1.equals(date2)
❌ Forgetting this() must be the first line

If you call this(...) inside a constructor, it MUST be the very first statement. Putting code before it is a compile error.

❌ equals() with wrong parameter type

Writing public boolean equals(Date other) — this overloads, not overrides! The correct signature is equals(Object object).

✅ Always match the Object class signature exactly, then downcast inside.
Topic 03
Encapsulation
Bundle data + methods together, and hide the internal details. Only expose what's necessary through a clean interface.
💡 Simple Explanation ▼

Encapsulation means: keep your data private and only let the outside world interact with it through controlled methods (getters/setters). This protects your object from invalid states.

🌍 Real-World Analogy
ATM machine. You don't get to open the machine and grab cash directly. You interact through a controlled interface (PIN, buttons). The bank's internal logic is hidden. You can only do what the machine allows.

Access levels in Java (from most to least restrictive): private → package/default → protected → public

  • private: Only inside the class itself
  • package (no modifier): Classes in the same package
  • protected: Same package + subclasses
  • public: Everyone
📝 Bad vs Good Encapsulation ▼
❌ Bad — Public fields
class Date {
  // Anyone can change this!
  public String month;
  public int day;
  public int year;
}

// Nothing stops invalid data:
d.day = -99;
d.month = "Octember";
✅ Good — Private + getters
class Date {
  private String month;
  private int day;

  public void setDay(int d) {
    // Validate before setting!
    if (d >= 1 && d <= 31)
      day = d;
  }
}
⚠️ Common Mistakes ▼
❌ Returning mutable objects directly from getters

If you have a private Date field and return it directly from a getter, the caller can modify the internal object — a privacy leak!

✅ Return a deep copy: return new Date(this.hireDate);
❌ Confusing protected with private

In derived classes, you cannot access private fields by name — even through inheritance. You must use the public/protected getters.

Topic 04
Inheritance
A derived class automatically gets everything from its base class. Write shared code once, reuse everywhere.
💡 Simple Explanation ▼

Inheritance lets you create a new class (child/derived/subclass) based on an existing class (parent/base/superclass). The child automatically has all public/protected fields and methods of the parent.

🌍 Real-World Analogy
Vehicles. Every vehicle has wheels, a speed, and can move. A Car is a Vehicle but also has doors and a trunk. A Truck is also a Vehicle but with a cargo bed. You define Vehicle once, and Car/Truck inherit all of it automatically.

Java uses extends for inheritance. Java only allows single inheritance (one parent class), but a class can implement multiple interfaces.

The "is a" relationship: An object of type FormattedDate is also a Date. You can assign it to a Date variable.

☕ Key Rules ▼
  • super(...) calls the parent constructor — must be the first line in child constructor
  • If you don't call super(), Java automatically calls the parent's no-arg constructor (compile error if it doesn't exist)
  • Private fields of parent are inherited but NOT directly accessible by name in child
  • super.methodName() calls the parent's version of an overridden method
  • final on a method = cannot be overridden. final on a class = cannot be extended.
📝 FormattedDate extends Date ▼
FormattedDate.java — Best practice version
public class FormattedDate extends Date {

    // New enum only in FormattedDate (not in Date)
    enum Format { DDMMYYYY, MMDDYYYY, YYYYMMDD }

    enum Month {
        JANUARY, FEBRUARY, MARCH, APRIL, MAY, JUNE,
        JULY, AUGUST, SEPTEMBER, OCTOBER, NOVEMBER, DECEMBER;
        private String index;
        Month() {
            int val = ordinal() + 1;
            index = val < 10 ? "0" + val : "" + val;
        }
        public static String dayToString(int val) {
            return val < 10 ? "0" + val : "" + val;
        }
    }

    private Format format; // NEW field — not in parent

    // Constructor: call super() first to init inherited fields
    public FormattedDate(String month, int day, int year, Format format) {
        super(month, day, year); // MUST be first line
        this.format = format;
    }

    // Convenience constructor — default format
    public FormattedDate(String month, int day, int year) {
        this(month, day, year, Format.YYYYMMDD);
    }

    // Default constructor
    public FormattedDate() {
        this("January", 1, 0, Format.YYYYMMDD);
    }

    // Static: format any Date with any Format
    public static String printInfo(Date date, Format f) {
        // Can't access date.month directly (it's private in Date!)
        // Must use public getters: date.getMonth(), date.getDay(), date.getYear()
        String mm = Month.valueOf(date.getMonth().toUpperCase()).index;
        String dd = Month.dayToString(date.getDay());
        switch (f) {
            case DDMMYYYY: return dd + "/" + mm + "/" + date.getYear();
            case MMDDYYYY: return mm + "/" + dd + "/" + date.getYear();
            case YYYYMMDD: return date.getYear() + "/" + mm + "/" + dd;
        }
        return null;
    }

    // Override toString() from Object (via Date)
    public String toString() {
        return printInfo(this, format);
    }

    // Override nextYear() — must return FormattedDate, not Date
    public FormattedDate nextYear() {
        return new FormattedDate(getMonth(), getDay(), getYear() + 1, format);
    }

    // Override equals — check parent first, then own fields
    public boolean equals(Object object) {
        return super.equals(object) && format == ((FormattedDate) object).format;
    }
}
⚠️ Common Mistakes ▼
❌ Accessing private parent fields directly

In FormattedDate, writing this.month fails because month is private in Date. The field is inherited, but the name is not accessible.

✅ Use getters: getMonth(), getDay(), getYear()
❌ Not calling super() in child constructor

If you forget super(month, day, year), Java calls the no-argument parent constructor. If none exists, it's a compile error. Even if it works, the parent's fields won't be initialized correctly.

❌ Thinking private means "not inherited"

Private fields are inherited — they exist in the object. They just can't be accessed by name from child code. Use getters/setters.

Topic 05
Polymorphism
One method name, many behaviours. Java decides which version to call at runtime based on the actual object type — not the variable type.
💡 Simple Explanation ▼

Polymorphism means "many forms." The same method call can behave differently depending on which object is actually there at runtime.

🌍 Real-World Analogy
The word "speak." If you tell a Dog to speak, it barks. If you tell a Cat to speak, it meows. If you tell a Person to speak, they talk. The command is the same — speak() — but the result depends on who you're talking to. That's polymorphism.

Late binding (dynamic binding): Java waits until runtime to decide which method version to call. It's based on the actual object type, not the variable's declared type.

Static binding (no late binding) applies to: private methods, final methods, and static methods.

☕ Upcasting vs. Downcasting ▼

Upcasting (safe, automatic): Assign a child object to a parent variable.

Downcasting (risky, manual): Cast a parent variable back to a specific child type. May throw ClassCastException at runtime if wrong.

Always use instanceof before downcasting to be safe.

📝 Polymorphism in Action (A, B, C hierarchy) ▼
Late binding demo from lectures
class A {
    public void printClassName() { System.out.println("A"); }

    // static: NO late binding — resolved at compile time by variable type
    public static void printStatic() { System.out.println("A static"); }

    // final: NO late binding — cannot be overridden
    public final void printFinal() { System.out.println("Always A"); }

    // private: NOT inherited — not overridable
    private void printPrivate() { System.out.println("A private"); }

    public String toString() { return "A"; }
}

class B extends A {
    public void printClassName() { System.out.println("B"); } // overrides A
    public static void printStatic() { System.out.println("B static"); } // hides A's static
    public String toString() { return "B"; }
}

class C extends B {
    public void printClassName() { System.out.println("C"); }
    public String toString() { return "C"; }
}

// Key test: variable type vs actual object type
A b2 = new B();  // variable type = A, actual object = B
b2.printClassName();  // OUTPUT: "B" ← Late binding uses ACTUAL type
b2.printStatic();     // OUTPUT: "A static" ← Static uses VARIABLE type
b2.printFinal();      // OUTPUT: "Always A" ← final, cannot be overridden
System.out.println(b2); // OUTPUT: "B" ← toString() uses late binding

// Safe downcasting with instanceof
if (b2 instanceof B) {
    B realB = (B) b2;  // safe downcast
}
⚠️ Common Mistakes ▼
❌ Thinking static methods use late binding

Static methods are NOT polymorphic. A b2 = new B(); b2.printStatic() calls A's version because the variable is of type A. This confuses many students.

✅ Rule: Static = resolved at compile time by variable type. Instance = resolved at runtime by object type.
❌ Unsafe downcasting without instanceof

B b = (B) someObject; — if someObject is actually an A, this throws ClassCastException at runtime.

✅ Always check first: if (someObject instanceof B) { B b = (B) someObject; }
❌ Calling child-specific methods on parent variable

A b2 = new B(); b2.printFromB(); — compile error! The variable type is A, so Java only knows about A's methods at compile time.

✅ Downcast first: ((B)b2).printFromB();
Topic 06
Abstract Classes
Define the structure and some behaviour, leave the rest for subclasses to fill in. Cannot be instantiated directly.
💡 Simple Explanation ▼

An abstract class is a class that says "I know I need a method called X, but I don't know how to implement it yet." Subclasses must provide the implementation.

🌍 Real-World Analogy
An architectural blueprint for "a building." The blueprint says every building must have a foundation, walls, and a roof — but it doesn't decide how they look. A HouseBlueprint and OfficeBlueprint each provide the concrete details. You can't build "a building" from the abstract plan alone.
  • Declared with abstract keyword on the class
  • Can have abstract methods (no body, just signature)
  • Can also have fully implemented (concrete) methods
  • Cannot be instantiated: new AbstractClass() = compile error
  • Abstract methods cannot be private, static, or final
  • A subclass must implement ALL abstract methods, or also be abstract
📝 Abstract Class Example ▼
AbstractClass + ChildClass from lectures
// Cannot instantiate this directly
public abstract class AbstractClass {

    // No body — subclass MUST implement this
    public abstract void doSomething();

    // Concrete method — inherited normally
    public void commonBehavior() {
        System.out.println("Shared by all subclasses");
    }
}

// Concrete class — must implement doSomething()
class ChildClass extends AbstractClass {
    public void doSomething() {
        System.out.println("ChildClass does the thing!");
    }
}

// Usage:
// AbstractClass a = new AbstractClass(); // COMPILE ERROR
AbstractClass a = new ChildClass(); // OK — variable can be abstract type
a.doSomething();     // "ChildClass does the thing!" (late binding)
a.commonBehavior(); // "Shared by all subclasses"

// Real example from Employee hierarchy:
public abstract class Employee {
    private String name;

    // Every employee can get paid — but HOW depends on type
    public abstract double getPay();

    // This uses getPay() — works via polymorphism
    public boolean samePay(Employee other) {
        return this.getPay() == other.getPay();
    }
}
⚠️ Common Mistakes ▼
❌ Trying to instantiate an abstract class

new AbstractClass() is a compile error, even if the abstract class has a constructor. It can only be created via a subclass.

❌ Forgetting to implement all abstract methods

If a subclass doesn't implement ALL abstract methods, it must also be declared abstract — or the compiler will error.

❌ Declaring abstract methods as private, static, or final

These modifiers contradict the purpose of an abstract method (which must be overridden). Compile error.

✅ Abstract methods must be public or protected.
Topic 07
Interfaces
A contract that any class can sign. Defines what a class must be able to do — without saying how.
💡 Simple Explanation ▼

An interface is like a contract. It lists methods that any implementing class MUST provide. Unlike abstract classes, a class can implement multiple interfaces.

🌍 Real-World Analogy
USB standard. The USB interface says "any device that plugs in must accept power and transfer data." A keyboard implements it. A mouse implements it. A flash drive implements it. Each does it differently, but they all fulfill the contract.
  • Declared with interface keyword, implemented with implements
  • All methods in an interface are implicitly public abstract
  • All variables are implicitly public static final
  • Java 8+: interfaces can have default methods (with body)
  • Java 8+: interfaces can have static methods
  • A class can implement multiple interfaces (unlike extending classes)
  • Interface is a type — you can have variables of interface type
📝 Interface Example ▼
Interface + Implementation
// Define the contract
public interface Rotatable {
    // Implicitly public abstract — no body!
    void rotateBy90Degrees();
    void rotateBy180Degrees();
}

// Implement multiple interfaces
public class Rectangle implements Rotatable {
    private int width, height;

    public void rotateBy90Degrees() {
        int temp = width;
        width = height;
        height = temp;
    }

    public void rotateBy180Degrees() {
        // Width and height stay the same for 180°
    }
}

// Interface as a type — polymorphism!
Rotatable r = new Rectangle(4, 6);
r.rotateBy90Degrees(); // Late binding — calls Rectangle's version

// Interface with constants (from lectures)
public interface MonthNumbers {
    // Implicitly public static final
    int JANUARY = 1, FEBRUARY = 2, MARCH = 3;
    // ... etc
}
⚠️ Common Mistakes ▼
❌ Implementing an interface but forgetting a method

If you implements an interface, you MUST provide all its methods — or your class must be declared abstract.

❌ Two interfaces with conflicting constant names

If two interfaces both define int ANSWER with different values, implementing both causes a compile error.

❌ Confusing interface and abstract class

Abstract classes can have instance variables and method bodies. Interfaces (traditionally) cannot have instance variables and all methods are abstract by default. A class can only extend ONE class, but implement MANY interfaces.

Topic 08
Cloneable & Comparable
Two essential Java interfaces: one for making safe copies of objects, one for defining a natural ordering for sorting.
💡 Comparable Interface ▼

Comparable<T> gives your class a natural ordering. Implement it to make objects sortable with Collections.sort().

One method to implement: public int compareTo(T other)

  • Returns negative if this comes before other
  • Returns zero if they are equal
  • Returns positive if this comes after other
🌍 Real-World Analogy
Alphabetical filing. When sorting files, "Apple" comes before "Banana". compareTo is like asking "which file comes first?" — negative = I come first, positive = you come first, zero = same position.
📋 Cloneable Interface ▼

Cloneable is a marker interface — it has no methods. It just signals to Object.clone() that copying is allowed.

The default Object.clone() does a shallow copy (bit-by-bit copy). This causes privacy leaks for fields that are reference types (objects, arrays).

You must override clone() to do a deep copy: call super.clone(), then manually copy any mutable object fields.

📝 Date with Comparable & Cloneable ▼
Date.java — March31 version from lectures
public class Date implements Comparable<Date>, Cloneable {

    private String month;
    private int day, year;

    // Clone: returns an independent copy of this Date
    public Date clone() {
        try {
            return (Date) super.clone(); // Safe for primitive fields
        } catch (CloneNotSupportedException e) {
            return null; // Won't happen since we implement Cloneable
        }
    }

    // compareTo: defines natural ordering (year → month → day)
    public int compareTo(Date that) {
        if (this.year != that.year)
            return this.year - that.year; // Earlier year = smaller

        // Same year: compare months by ordinal
        int m1 = FormattedDate.Month.valueOf(this.month.toUpperCase()).ordinal();
        int m2 = FormattedDate.Month.valueOf(that.month.toUpperCase()).ordinal();
        if (m1 != m2) return m1 - m2;

        return this.day - that.day; // Same year+month: compare day
    }
    // ... (rest of fields, constructors, getters — same as before)
}

// DateHolder: shows why deep copy matters
class DateHolder implements Cloneable {
    private int n;
    private String s;
    private Date date; // mutable object — needs deep copy!

    public DateHolder(int n, String s, Date d) {
        this.n = n; this.s = s;
        this.date = d.clone(); // Defensive copy in constructor
    }

    public DateHolder clone() {
        try {
            DateHolder copy = (DateHolder) super.clone(); // shallow first
            copy.date = date.clone(); // then deep copy the Date field
            return copy;
        } catch (CloneNotSupportedException e) { return null; }
    }
}

// Sorting with Comparable:
ArrayList<FormattedDate> list = new ArrayList<>();
// ... add dates ...
Collections.sort(list); // Uses compareTo() automatically
⚠️ Common Mistakes ▼
❌ Shallow copy of objects containing references

super.clone() copies primitive fields correctly but copies references as-is. Two objects end up sharing the same inner object — a privacy leak.

✅ After super.clone(), clone each mutable reference field individually.
❌ Implementing Cloneable but not overriding clone()

The Cloneable interface has no methods — so if you only implement it but don't override clone(), you get the protected Object clone() which may not be accessible or correct.

❌ compareTo returning wrong sign

If you want ascending order (smallest first), compareTo should return negative when this is smaller. Returning the wrong sign reverses your sort order.

✅ Test: if a.compareTo(b) < 0, then a should come before b in sorted order.

🎯 Top 10 Things to Know for Exams ▼
  1. == vs .equals(): == compares memory addresses. Use .equals() for value comparison of objects (Strings, Dates, etc.)
  2. super() must be first: In any constructor, calling super() or this() must be the very first statement — no exceptions.
  3. private stays private: Even in subclasses, private fields can't be accessed by name. Use getters.
  4. Late binding = actual object type: When you call an instance method, Java looks at what the object actually IS — not what variable type holds it.
  5. Static = compile time: Static methods are decided at compile time based on the variable's declared type, not the actual object.
  6. instanceof before downcasting: Always check if (obj instanceof ChildClass) before casting to avoid ClassCastException.
  7. Abstract class ≠ interface: Abstract class can have fields + concrete methods. Interface (classically) cannot have instance variables.
  8. Enum ordinal is 0-based: JANUARY.ordinal() == 0. Add 1 to get the month number.
  9. Deep copy for mutable fields: When cloning, always clone any fields that are objects (otherwise two objects share the same inner object).
  10. compareTo contract: negative = comes before, zero = equal, positive = comes after. Wrong sign = wrong sort order.
